# 文件夹手动删除或者清理
## 特定文件之外的文件(如工作目录只保留data.json文件)
```bash
find . -type f ! -name "data.json" -exec rm -f {} \;
# linux 如何rm删除某个目录下除了data.json外的所有文件和文件夹
## 方法 1：使用 `find` + `rm`（推荐）
cd /path/to/directory  # 进入目标目录
find . -mindepth 1 -maxdepth 1 ! -name "data.json" -exec rm -rf {} +
## 方法 2：使用 `find` + `xargs`
cd /path/to/directory  # 进入目标目录
find . -mindepth 1 -maxdepth 1 ! -name "data.json" | xargs rm -rf
## 方法 3：使用 `ls` + `grep` + `xargs`（不推荐，仅适用于简单场景）
cd /path/to/directory  # 进入目标目录
ls -A | grep -v "data.json" | xargs rm -rf
```

## s3数据有问题
## 查找问题
1.  下载s3上的数据
	1. 如果不能下载，那么就是上传s3出现了问题(大概率是块丢失了)
	2. 如果能下载，接着检测数据md5值和大小是不是和数据库保持一致
2.  获取下载数据的md5值和文件大小与mongo库中存储的记录做对比
	1.  如果有差异，那么就是上传s3出现了问题
	2. 如果保持一致，那么检测R1和R2是不是reads数保持一致
3.  通过seqkit stats子命令检测数据
	1. 如果能运行出结果且R1和R2数据量一致，那么这个数据大概率没问题，需要查下是不是分析的问题或者下载s3时候存储有问题
	2. 如果运行报错或者R1和R2数据量不一致，那么数据不是标准的fastq.gz，可能是拆分或者合并出错
## 产生原因
- 网络波动或者s3出现问题导致某些上传块有问题
- 本地存储问题引起的IO问题导致合并出错
- 历史有些拆分数据可能拆分数据量不一致但是当时没有报错
## 解决方案
-  如果是s3上传块有问题，找到当初的工作流重运行，ftm会自动覆盖
-  如果是合并导致的问题，重新运行合并流程
-  如果是拆分导致的问题
	- 如果是近期的，可以进行重新拆分再确认使用进行流转
	- 如果太过久远，bcl文件已经不存在了，那么就没有办法了

```ini
[global]
   workgroup = WORKGROUP          # 与 Windows 工作组保持一致
   server string = Samba Server
   security = user                # 需要用户名密码登录
   map to guest = never
   dns proxy = no

[share]                           # 共享名称（Windows 看到的名字）
   comment = Shared Folder
   path = /data/share             # Linux 上的实际路径
   browseable = yes               # 可在网络中浏览
   read only = no                 # 允许写入（改为 yes 则只读）
   writable = yes
   valid users = @sambashare      # 允许访问的用户组
   create mask = 0664             # 新建文件权限
   directory mask = 0775          # 新建目录权限
```
